📑 Master Spec v2.1: Cloud Architecture, Ethics & Privacy
Foco: Descentralização (Cloud), Monitoramento e Privacidade de Dados.

1. Migração para a Nuvem (Cloud vs. Localhost)
O banco de dados não pode ser localhost. Para o sistema funcionar no curso, na sua casa ou no celular do aluno, ele deve ser hospedado em um serviço de nuvem.

Configuração para o Agente:
Substituir o PostgreSQL local por uma instância gerenciada (Ex: Supabase, Render ou Railway).

Vantagem: O banco fica online 24/7.

Confiabilidade: Backups automáticos e SSL (criptografia) nativo.

2. Monitoramento Ético (O seu Painel de Gestão)
Como você cobrará por aluno ativo, você não precisa (e não deve) ver as senhas ou dados sensíveis de navegação dos alunos. O seu acesso será de SuperAdmin (SaaS Owner).

Funcionalidades do seu Monitor:
Contador de Alunos Ativos: Um dashboard que mostra apenas: "Total de Alunos: 40".

Status de Integração: Um log técnico dizendo se o Webhook do Mercado Pago está funcionando.

Cálculo de Invoice: Um gerador automático da sua fatura para o curso (Ex: 40 alunos x R$ 25,00 = R$ 1.000,00).

3. Privacidade e Segurança (Hashing)
Para manter a ética, o sistema utilizará One-Way Hashing (Argon2id ou BCrypt).

Como funciona: Quando o ADM do curso cria uma senha para o aluno, o sistema transforma "senha123" em uma string gigante e ilegível.

Ética: Nem você, nem o dono do curso conseguem "ver" a senha no banco de dados. Só é possível validar se a senha digitada está correta.

4. Banco de Dados Atualizado (PostgreSQL)
O agente deve implementar as tabelas garantindo que as permissões de acesso sejam separadas:

SQL
-- Adicionando controle de quem é o dono do sistema (VOCÊ) vs Dono do Curso (ADM)
ALTER TABLE users ADD COLUMN role VARCHAR(20) DEFAULT 'student'; 
-- Roles: 'superadmin' (você), 'admin' (curso), 'student' (aluno)

-- O SuperAdmin (você) tem acesso apenas à tabela de métricas de uso para cobrança.
5. Instruções Adicionais para o Agente
Deploy Readiness: Prepare o código para ser enviado ao Render.com ou Railway.app. O sistema deve ler as credenciais do banco via Variáveis de Ambiente (.env), nunca fixas no código.

Separação de Contexto: - O Dono do Curso vê: Nomes, CPFs e quem pagou.

Você (SaaS Owner) vê: Quantidade de usuários e logs de erro do sistema.

O Aluno vê: Apenas suas próprias faturas.

Logs de Auditoria: Implemente um sistema de log que registre quando um pagamento foi aprovado, para que você possa provar o valor da sua cobrança mensal.

Por que isso resolve seu problema?
Distância: Sendo Cloud, o curso acessa de lá e você monitora de São Luís sem problemas.

Confiabilidade: O uso de serviços como Supabase ou Render passa a imagem de uma empresa de software profissional, não um projeto de "estudante".

Ética: Ao usar Hashing e Roles (papéis de usuário), você protege os dados dos alunos e foca apenas no que importa para o seu negócio: monitorar o volume de uso para emitir sua cobrança de R$ 25/aluno.
