Aqui está o Master Prompt atualizado e consolidado. Este arquivo foi desenhado para ser entregue a uma IA de desenvolvimento (como o agente do Antigravity, Cursor ou Claude) para que ela construa o ecossistema completo: Frontend, Backend, Painel Administrativo e Integração Real com Mercado Pago.

📑 Especificação Técnica: Portal Financeiro LED v2.0
Contexto: Sistema de gestão de mensalidades com automação de pagamentos.
Stack Principal: Go (Backend), PostgreSQL (Banco de Dados), Tailwind/JS (Frontend).
Gateway: Mercado Pago (Checkout Pro + Webhooks).

1. Escopo de Funcionalidades
A. Painel do Aluno (Interface Stripe-Style)
Login: Autenticação via CPF e Senha.

Dashboard:

Saudação personalizada: "Bem-vindo, {nome}!".

Ficha Financeira: Lista de faturas (Nome, Vencimento, Status, Valor).

Botão Pagar: Integração direta com o Checkout Pro do Mercado Pago.

Relatório IR: Botão para gerar resumo anual de pagamentos.

Sistema de Notificação:

Se a fatura vencer em menos de 7 dias, exibir alerta visual: "Atenção: Sua fatura vence em breve!".

Regra de Negócio (Multa): Se a data atual > data de vencimento, o sistema deve atualizar o valor automaticamente de R$ 300,00 para R$ 500,00.

B. Painel Administrativo (ADM)
Gestão de Usuários: Criar, editar e excluir alunos.

Gestão de Senhas: Gerar e resetar senhas para fornecer aos alunos (armazenadas em Argon2id).

Gestão Financeira: Visualizar quem pagou e quem está inadimplente.

Sincronização: Botão para importar/atualizar lista de alunos e faturas via CSV.

2. Arquitetura de Dados (PostgreSQL)
O banco de dados será o PostgreSQL, estruturado para garantir integridade e auditoria:

SQL
-- Tabela de Alunos
CREATE TABLE users (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    cpf VARCHAR(11) UNIQUE NOT NULL,
    password_hash TEXT NOT NULL,
    full_name VARCHAR(255) NOT NULL,
    email VARCHAR(255),
    is_admin BOOLEAN DEFAULT FALSE,
    created_at TIMESTAMP DEFAULT NOW()
);

-- Tabela de Faturas
CREATE TABLE invoices (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID REFERENCES users(id),
    description VARCHAR(255),
    amount DECIMAL(10,2) DEFAULT 300.00,
    due_date DATE NOT NULL,
    status VARCHAR(20) DEFAULT 'pending', -- 'pending', 'paid', 'overdue'
    mp_preference_id TEXT, -- ID retornado pelo Mercado Pago
    paid_at TIMESTAMP
);
3. Integração Mercado Pago (Fluxo Real)
Requisitos de Credenciais (Solicitar ao Usuário):
Para a API funcionar, o agente deve configurar os seguintes campos no arquivo .env:

MP_ACCESS_TOKEN: Token de produção da conta LED.

MP_PUBLIC_KEY: Chave pública para o frontend.

WEBHOOK_SECRET: Para validar a autenticidade das notificações do Mercado Pago.

Fluxo do Webhook:
O Backend deve expor a rota POST /api/v1/webhooks/mercadopago.

Recebe o ID do pagamento.

Consulta a API do Mercado Pago para verificar se o status é approved.

Se aprovado, o sistema altera o status da fatura para paid e registra o paid_at.

4. Lógica de Negócio e Segurança (Hardening)
Cálculo Dinâmico de Valor: O Backend, ao dar o GET nas faturas, deve comparar due_date com current_date. Se current_date > due_date E status != 'paid', o amount deve ser retornado como 500.00.

Segurança de Sessão: Uso de JWT para alunos e ADM, com permissões (RBAC) para garantir que alunos não acessem o painel administrativo.

Sanitização de Inputs: Proteção contra SQL Injection no campo de busca de CPF.

5. Instruções para o Agente de Desenvolvimento
Backend (Go): - Crie as rotas de login, dashboard de faturas e geração de preferência do Mercado Pago.

Implemente a lógica de multa (300 -> 500) no momento da consulta e antes de gerar a preferência de pagamento.

Frontend: - Use Tailwind CSS para replicar a estética limpa da Stripe.

Implemente o estado de alerta (notificação) para faturas com menos de 1 semana de vencimento.

Painel ADM:

Crie uma área restrita /admin para gestão de senhas e monitoramento de recebíveis.

Atenção Agente: Solicite ao usuário as credenciais do Mercado Pago assim que a estrutura base do servidor estiver pronta para teste de integração.
